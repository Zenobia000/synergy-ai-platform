# PRD — AI 直銷 Care Copilot（Phase I MVP）

> 功能模組導向說明文件。深技術規格見 00_tech-spec / 01_architecture；資料模型見 03_data-model；API 見 02_api。

---

## 1. 一句話定位

**「直銷商的關懷 Copilot — 記得每位你關心的人、知道今天該聯絡誰、講話有溫度不像在推銷。」**

| 指標 | 定義 | Pilot 門檻 |
| :--- | :--- | :--- |
| **PMF** | W4 結尾，5 位教練裡至少 4 位主動說願意付 USD 39/月 | 4/5 |
| **北極星** | 每位活躍直銷商每週的 Care Action 數（關懷動作，非推銷） | 持續成長趨勢 |
| **Pilot 規模** | 5 位 Synergy 教練 + 1 位 Leader + 5 位下線，為期 4 週 | — |

---

## 2. 目標使用者

| Persona | 一句話 |
| :--- | :--- |
| **Amy**（主 Persona，Active 直銷商） | 帶 20–100 位客戶的中段直銷商，需要記住每個人的情境、拿捏朋友 tone，並在對話現場接住異議。 |
| **Nina**（次 Persona，Newbie 直銷商） | 剛入行 6 個月以內的新手，需要每日節奏引導、有模板可參考但不死板，以及合規防護。 |
| **Linda**（次 Persona，Leader） | 帶 10–50 位下線的資深 Leader，需要個人客戶工具加上下線招募漏斗可視化，確保下線不踩線。 |
| **Tina**（GTM Persona，Top Influencer） | 業界 KOL，不是主要使用者，而是 GTM 病毒引擎：代言推薦帶動教練圈擴散，以 Founders Circle 換長期綁定。 |

| 模組 | 一句話功能 | 主要使用者 | AI 引擎 |
| :--- | :--- | :--- | :--- |
| **T01 關係記憶活檔案** | 給每位客戶建一份隨時可補充的活檔案：健康關注、家庭、興趣、互動脈絡。 | Amy / Nina | Sonnet 4.6（insight 生成）|
| **T02 生活事件雷達** | 偵測生日、換工作、寶寶誕生等生活事件，推出非推銷的關懷提醒。 | Amy | Haiku 4.5（文字抽取）|
| **T03 語氣情緒感測** | 讀對方近期訊息判斷「壓力大／中性／開心」三檔，告知今天適不適合提產品。 | Amy / Nina | Haiku 4.5（三檔分類）|
| **T04 太業務員警報** | 連 3 則推銷就觸發警告，預生純關懷草稿，保住對話溫度。 | Amy / Nina | 純規則引擎（無 AI）|
| **T05 今日 5 件事** | App 首頁每日 5 張優先任務卡，早上 10 秒知道今天該關心誰。 | Amy / Nina | 規則排序（v1 無 AI）|
| **T06 訊息草稿** | 依活檔案脈絡生 3 種語氣草稿（關懷／隨意／商業），AI 不自動發送。 | Amy / Nina | Haiku 4.5 首字 + Sonnet 4.6 全文 streaming |
| **T07 樣品追蹤** | 發出樣品後自動排 48h／72h／7d 三個跟進提醒並預生草稿。 | Amy | Haiku 4.5（跟進草稿）|
| **T08 語音草稿** | 文字草稿一鍵轉 TTS 語音（≤ 60 秒），直銷商下載後自行傳給客戶。 | Amy | VoiceProvider 抽象（OpenAI TTS 或 ElevenLabs，W4 選型）|
| **T09 快速異議處理** | 10 種常見異議一鍵獲得共情型／提問型／邀請型 3 種接話建議。 | Amy / Nina | Haiku 4.5（速度優先）|
| **T10 健康問卷**（可選） | 對高意願客戶發 7 天有效一次性連結，填完後 AI 生摘要回填活檔案。 | Amy | Sonnet 4.6（問卷摘要）|
| **T11 招募漏斗** | 追蹤暖名單→曝光→邀約→簽約四階段轉換率，招募話術走最嚴格合規。 | Linda | Sonnet 4.6 |
| **SYS 合規低語** | 所有外送草稿必過 regex sidecar（< 50ms），綠／黃／紅三燈，紅燈強制阻擋。 | 全部 | 純規則引擎（無 AI）|
| **PLT 平台支撐** | 隱私同意、Freemium 配額、Care Streak、病毒擴散、學習紀錄基礎架構。 | 全部 | 無 AI（純寫入）|
| **LINE OA 收發** | 客戶傳訊 LINE OA → 系統收訊驗簽 → 新客輪派 → AI 輔助草稿 → 教練審核 → 按發送；AI 不自動回。 | Amy / Linda | Haiku 4.5（回覆草稿）|
| **多教練輪派** | 新客戶 round-robin 輪派下一位 active 教練；既有客戶 sticky；Leader 可看全租戶並改派。 | Linda | 純規則（無 AI）|

## 4. 各模組說明

### T01 關係記憶活檔案

每位客戶擁有一份隨時可累積的活檔案，涵蓋健康關注、家庭背景、興趣愛好與歷次互動摘要。直銷商可透過四種方式補充資料：手動輸入、截圖上傳、語音備忘、問卷結果自動回填。解決的痛點是「帶超過 30 人就記不住每個人的狀況」，讓 Amy 每次開口都有脈絡依據。

### T02 生活事件雷達

當活檔案或貼上的對話文字出現生日、升遷、換工作、生寶寶等生活事件關鍵詞，系統自動觸發一張非推銷的關懷提醒任務卡。日期型事件（如生日）走純規則偵測；文字型事件走 Haiku 4.5 抽取。解決「朋友有大事發生卻沒注意到，沒有關心到位」的問題。

### T03 語氣情緒感測

讀取教練貼入的近期對話紀錄，以 Haiku 4.5 判斷客戶目前情緒屬於「壓力大／中性／開心」三檔，並在草稿頁面顯示今天的溝通建議。若情緒為「壓力大」，系統自動鎖定草稿語氣為「關懷型」並隱藏商業型草稿，保護關係不被過早推銷傷害。

### T04 太業務員警報

純規則引擎偵測同一對話中連續出現 3 則以上含產品關鍵字或購買連結的訊息，立即觸發橘色警示卡，並預生一則純關懷草稿供教練改口使用。解決「知道要收手但不知道說什麼」的行為慣性問題。

### T05 今日 5 件事

App 首頁每日凌晨由排程器重新計算，按優先規則列出最多 5 張任務卡，包含：樣品跟進到期、生活事件提醒、LINE 待回覆訊息、招募漏斗進度提醒，以及系統建議主動關心的客戶。教練早晨打開 App 10 秒即可知道今天要做什麼，完成後標記清除。

### T06 訊息草稿

教練選定聯絡人後，系統拉取活檔案脈絡，以 Haiku 4.5 生成首字（< 500ms）讓教練感覺「開始了」，再由 Sonnet 4.6 接力完成三種語氣的完整草稿（關懷型、隨意型、商業型）。全文經合規掃描後方可複製使用，AI 永不自動發送。

### T07 樣品追蹤

教練登錄寄出樣品後，系統自動排定三波 APScheduler 提醒任務（48h、72h、7 天），並提前以 Haiku 4.5 預生跟進草稿，直接掛入今日 5 件事任務卡的「查看草稿」快捷入口。解決「給了樣品就忘了追蹤」的常見問題。

### T08 語音草稿

教練確認文字草稿後，可一鍵呼叫 TTS 合成 ≤ 60 秒的語音檔，下載後自行傳送給客戶（語音訊息比文字更有溫度）。語音檔存放於 GCS，7 天後自動刪除。TTS 供應商（OpenAI TTS 或 ElevenLabs）以 VoiceProvider 介面抽象，W4 選型前可自由切換。

### T09 快速異議處理

預設 10 種直銷常見異議（如「太貴了」「先考慮看看」「我不需要」），教練遇到時一鍵呼叫 Haiku 4.5 生成共情型、提問型、邀請型三種接話建議，速度目標 < 2 秒。解決「對方提出異議，當場不知道怎麼接」的臨場尷尬。

### T10 健康問卷（可選）

教練對已建立活檔案且表現出高意願的客戶，生成一個 7 天有效的一次性填寫連結，客戶無需登入即可填寫 12 題健康問卷。填寫完成後由 Sonnet 4.6 生成摘要並自動回填活檔案中的健康關注欄位，同時產生一張今日任務卡提醒教練查看摘要。

### T11 招募漏斗

Linda 等 Leader 可在招募漏斗頁面追蹤每位潛在下線從暖名單→曝光→邀約→簽約的四階段進度，並查看轉換率。Sonnet 4.6 協助生成最嚴格合規等級的招募話術草稿（涉及 FTC 收入保證限制）。

### SYS 合規低語

所有外送類型草稿（訊息草稿、樣品跟進、語音腳本、異議回應、問卷摘要、招募話術）在被教練採用前，必須通過 regex sidecar（< 50ms，50 詞庫，無 AI）。結果分三燈：綠燈可用、黃燈警示可覆蓋、紅燈強制阻擋且必須改寫。每次掃描 100% 寫入合規紀錄。

### PLT 平台支撐

提供產品所需的底層基礎設施：使用者隱私同意記錄、Freemium／Pro／Pro Plus 三方案配額控管（聯絡人上限、每日草稿數、語音數）、AI 每日成本熔斷（每人 ≤ USD 0.30/日）、Care Streak 連續行動視覺化、推薦獎勵機制，以及 100% 寫入的 AI 操作學習紀錄（供 Phase II 模型訓練用）。

### LINE OA 收發

每個租戶對應一個 LINE 官方帳號（OA）。客戶傳訊給 OA 後，系統在 3 秒內完成 webhook 驗簽、聯絡人比對、新客戶輪派，並將收到的訊息寫入 `inbound_messages`，同時在教練的今日 5 件事推送「待回覆」任務卡。接著自動生成輔助回覆草稿（Haiku 4.5，過合規）並通知教練審核。教練在 App 確認草稿後按「發送」，系統呼叫 LINE Messaging API push message 送達客戶。AI 不自動回覆；合規紅燈時發送按鈕禁用。

### 多教練輪派

當 LINE OA 收到來自新客戶的訊息，系統以 round-robin 演算法找到下一位 `active` 狀態的教練並指派。既有客戶維持 sticky（訊息永遠路由至同一位教練）。Leader 可在 App 查看全租戶待回覆訊息並手動改派任一教練，改派後 `distributor_id` 及對應任務卡同步更新。

## 5. 範圍

### 做（Phase I MVP）

- 11 個核心工具（T01–T11）+ SYS 合規低語 + PLT 平台支撐
- Mobile PWA（iOS Safari 優先，Android Chrome 次之）
- LINE OA 收訊 + 多教練 round-robin 輪派 + 輔助回覆草稿 + 教練審核發送
- FastAPI 自建 JWT 認證（distributor / leader 兩角色）
- PostgreSQL 16 + pgvector 多租戶 RLS 隔離
- Freemium / Pro / Pro Plus 三種方案配額控管
- AI 每日成本熔斷（每人 ≤ USD 0.30）
- 觀測基礎設施（OTel + Langfuse + Sentry）
- 客戶資料請求處理（export / delete）
- 隱私同意記錄

### 不做（明確排除）

| 排除項目 | 說明 |
| :--- | :--- |
| LINE 自動回覆 / 主動群發 | AI 不自動回（硬約束 G.1 / G.6）；群發非個人化關懷，排除 |
| WhatsApp / Instagram 官方 API 對接 | v1 不在範疇；LINE OA 已納入 |
| 佣金計算與組織樹管理 | Phase II / III |
| 完整電商功能 | 超出 prosumer 工具範疇 |
| 品牌後台管理 UI | 資料模型 Day 1 已抽象，管理 UI 排 Y4+ |
| 多語系（繁中以外） | v1 繁中為主，語音支援英語 |
| 直銷商自錄語音變體 | v1 TTS 單向，不做錄音變體 |
| 健康問卷即時多版本 A/B 分析 | Phase II |
| 完整 Leader 下線管理儀表板 | Phase II，v1 只有招募漏斗統計 |
| Brand Admin 品牌後台 Persona | Y1–Y3 不在 scope |
| 非 LINE OA 客戶主動外發訊息 | 非 OA 客戶仍手動補資料 + 複製草稿至 LINE 外傳 |

## 6. 關鍵使用情境

### 情境 A — Amy 早晨開 App 處理今日任務

Amy 早上 9:00 打開 App，首頁今日 5 件事顯示：「Cathy 的 Omega+ 樣品已滿 48 小時」任務卡，草稿已預生好。Amy 看了一眼草稿，稍作修改，複製後貼到 LINE 傳給 Cathy，標記任務完成。整個流程不超過 2 分鐘。

### 情境 B — LINE 客戶來訊，輪派給教練並輔助回覆（完整 LINE 收發流程）

1. 新客戶「王小明」在 LINE 傳訊給 Synergy 教練團隊的 LINE 官方帳號：「我想多了解一下這個產品」
2. 系統 3 秒內完成：webhook 驗簽 → 確認王小明是新客戶 → round-robin 輪派給下一位 active 教練 Amy → 建立聯絡人活檔案（source = line_oa）→ 寫入 inbound_messages
3. Amy 的 App 出現今日任務卡：「LINE 新客戶王小明，待回覆」
4. Amy 點開任務卡，看到 AI 預生的輔助回覆草稿（已過合規，顯示綠燈）
5. Amy 修改幾個字，按「發送」按鈕
6. 系統再次合規確認（綠燈）→ 呼叫 LINE Messaging API push message → 王小明在 LINE 上收到 Amy 以 OA 名義發出的回覆

### 情境 C — Nina 遇到異議當場接住

Nina 正在和潛在客戶聊，對方說「這個太貴了，我不需要」。Nina 打開 App 的異議處理器，選「太貴」，Haiku 4.5 在 2 秒內給出三種接話：共情型「我懂，一開始我也覺得，不過後來我自己用了發現...」、提問型「你主要是擔心哪一塊費用？」、邀請型「先體驗看看就好，不一定要馬上決定」。Nina 選用提問型，繼續對話。

### 情境 D — 合規警示強制介入

Amy 在 LINE 連續傳了 3 則含產品詞彙的訊息後，App 彈出太業務員警報：「最近說了 3 次產品，先換個話題關心一下？」並預生一則純關懷草稿。Amy 本來要再推，看到提示後改用關懷草稿，保住了關係溫度。

### 情境 E — Linda 查看下線招募進度並改派 LINE 訊息

Linda 打開招募漏斗，看到本週 3 位暖名單從「曝光」推進到「邀約」階段，轉換率符合預期。她切換到 LINE 訊息管理頁，發現其中一則新客戶訊息被 round-robin 分派給了剛請假的 Kevin，Linda 手動改派給 Amy，訊息任務卡即時更新至 Amy 的今日 5 件事。

## 7. 成功標準與 Kill Criteria

### 成功標準（依 Persona）

| Persona | 關鍵驗證目標 |
| :--- | :--- |
| **Amy**（5 位教練） | W4 結尾至少 4/5 主動說願意付 $39；每週 Care Action 數 > 推銷數；草稿採用率有成長；整段 Pilot 0 次踩合規紅線 |
| **Nina**（2–3 位新手） | 7 天裡至少 5 天打開 App；W4 有 15 位以上活檔案；草稿採用率高於 Amy |
| **Linda**（1 位 Leader） | 個人願意付 $39；下線 5 人裡至少 3 人每週開 App ≥ 4 天；每週看招募漏斗 ≥ 2 次 |
| **Tina**（GTM） | 25 個邀請裡至少 10 個簽 Founders Circle；下線候補名單 ≥ 50 人 |

### Kill Criteria（任一觸發即停 Pilot 重評）

| 視角 | Kill 條件 |
| :--- | :--- |
| 系統 | 重大故障連續 > 24 小時 |
| Amy | AI 草稿採用率 < 30%；教練流失 ≥ 2/5；自願付 < 3/5 |
| Nina | 4 週後仍 < 1/3 維持每日節奏 |
| Linda | 下線 5 人中 < 2 人活躍 |
| 合規 | 任何 Persona 外送踩線 ≥ 3 次 |
| 成本 | AI 成本超預算 50%（含語音超量）|

## 8. 4 週 Roadmap

| 週次 | 交付重點 | 關鍵里程碑 |
| :--- | :--- | :--- |
| **W1** | T01、SYS 合規、JWT 認證、完整資料模型（含 LINE 相關表）、觀測基礎 | 後端骨架可跑，無 AI 草稿 |
