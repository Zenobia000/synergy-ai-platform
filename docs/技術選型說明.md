# 技術選型白話說明 — Care Copilot

> 這份文件把 `00_tech-spec.md` 與 `01_architecture.md` 裡的每一個技術選型，用**白話**講清楚三件事：
> **① 它是什麼（生活化比喻）　② 在我們系統裡負責什麼　③ 為什麼選它**。
> 不需要工程背景也能讀。看完你就知道每個名詞「是幹嘛的、為什麼在這」。

---

## 0. 先看整體：一句話搭起來

> **客戶從 `LINE 官方帳號` 傳訊息進來，系統收到後自動輪派給一位教練，並用 AI 生一則回覆草稿（先過合規關）；教練在 App 看到草稿、改好後按「發送」，系統才透過 `LINE Messaging API` 把訊息送出去——AI 不會自己回，一定要教練點確認。
> 使用者在手機上看到的畫面是 `React`，畫面由 `Vite` 打包、`Tailwind` 上妝、`PWA` 讓它像 App；
> 手機把指令送到後端大腦 `FastAPI`（用 `Python` 寫）；
> 資料全部放在自己跑的 `PostgreSQL` 資料庫（裝了向量搜尋外掛 `pgvector`），部署在 `GCP` 上；
> 需要「動腦」的事交給 `Claude` AI（快事用 `Haiku`、難事用 `Sonnet`），由 `LangGraph` 指揮；
> 每一句要送出去的話，先被 `合規 sidecar` 把關；
> 整個系統的健康狀況由 `OpenTelemetry / Langfuse / Sentry` 盯著。**

用一張對照表先建立印象，後面再逐個展開：

| 層 | 技術 | 一句話 |
| :--- | :--- | :--- |
| 前端（畫面） | React / Vite / Tailwind / PWA / Zustand / TanStack Query | 使用者手機上看到、點到的所有東西 |
| 連線 | REST + SSE | 手機跟後端講話的方式（SSE 讓草稿逐字浮現） |
| LINE 整合 | line-bot-sdk / LINE Messaging API / webhook | 收客戶 LINE 訊息、輪派教練、把教練核准後的回覆送出去 |
| 後端（大腦） | FastAPI / Python / uv / APScheduler / Alembic | 接收指令、處理邏輯、排定時提醒 |
| 資料與帳號 | PostgreSQL + pgvector / RLS / 自建 JWT / GCS | 存資料、管登入、做客戶資料隔離、存語音檔 |
| AI（智慧） | LangGraph / Claude Haiku / Claude Sonnet / TTS | 生草稿、判情緒、寫摘要、轉語音 |
| 守門 | 合規 sidecar / 太業務員規則引擎 | 送出前自動把關，避免踩線 |
| 監看 | OpenTelemetry / Langfuse / Sentry | 看系統健不健康、AI 花多少錢、哪裡出錯 |

## 1. 前端 — 使用者手機上的畫面

### React 19（前端框架）
- **是什麼**：蓋畫面的「鋼骨結構」。如果把 App 比喻成房子，React 就是讓你能用一塊塊「元件」（按鈕、卡片、清單）快速組裝、而且資料一變畫面就自動跟著變的工具。
- **作用**：今日 5 件事的任務卡、客戶詳情頁、草稿區……所有看得到的介面都用 React 元件拼出來。
- **為什麼選它**：市佔最高、人才最好找、生態最完整；19 版的新功能（Concurrent）讓畫面在資料更新時不卡頓，適合「草稿逐字浮現」這種即時體驗。

### Vite 6（前端建置工具）
- **是什麼**：前端的「工地機具 + 打包機」。開發時讓你存檔後畫面**瞬間**更新（不用等），上線前把程式壓縮打包成瀏覽器看得懂的檔案。
- **作用**：開發伺服器（`:3002`）、把 React/Tailwind 打包、串接 PWA 外掛。
- **為什麼選它**：比舊工具（Webpack）快非常多，開發體驗好；PWA 外掛生態成熟，剛好我們要做「像 App 的網頁」。

### Tailwind CSS v4（樣式工具）
- **是什麼**：幫畫面「上妝排版」的工具。不用另外寫一堆樣式檔，直接在元件上用簡短代號（如「間距 8」「圓角大」）就能調好外觀。
- **作用**：統一全 App 的顏色、字級、間距、圓角，對齊 Apple 風格。
- **為什麼選它**：開發快、風格一致、好維護；v4 改成「CSS 優先」，把設計 token（顏色/字級的中央定義）管得更乾淨——改一個地方，全站跟著變。

### PWA（vite-plugin-pwa，漸進式網頁應用）
- **是什麼**：讓「網頁」可以像「手機 App」一樣——能加到主畫面、有圖示、開得快、弱網也能用。
- **作用**：直銷商把 Care Copilot 加到 iPhone 主畫面，用起來跟原生 App 幾乎一樣；Service Worker 會快取畫面，開 App 不用空白等待。
- **為什麼選它**：不用上架 App Store、不用分別開發 iOS/Android，一套網頁兩邊通用，最省成本——對 4 週 Pilot 特別划算。

### Apple 設計系統 tokens（視覺風格）
- **是什麼**：一套「設計規格書」，把顏色、字體、圓角、陰影的標準值寫死，大家照著用。
- **作用**：讓整個 App 看起來像 iPhone 內建 App 一樣乾淨、有質感、好上手。
- **為什麼選它**：80% 直銷商用 iPhone，Apple 風格對她們最熟悉、最有信任感。

### Zustand（前端狀態管理）
- **是什麼**：畫面內部的「小白板」。把「目前選了哪個客戶」「側邊欄開著沒」這類暫存資訊集中放，讓不同元件都讀得到、不打架。
- **作用**：管理 App 內部的即時狀態（非伺服器資料），例如目前語氣選了「關懷」。
- **為什麼選它**：非常輕量、學習成本低、和 React 19 相容好；比老牌 Redux 簡單太多，對小團隊剛好。

### TanStack Query（資料抓取與快取）
- **是什麼**：前端的「資料管家」。負責跟後端要資料、把要過的資料先暫存起來（下次秒開）、資料變了自動重抓。
- **作用**：抓今日任務、客戶清單、草稿等；支援草稿的 SSE 串流；做「樂觀更新」（你一點完成，畫面先變，背景再跟後端確認）。
- **為什麼選它**：把「載入中 / 失敗 / 重試 / 快取」這些麻煩事全包了，少寫很多重複程式碼，畫面也更順。

### SSE（Server-Sent Events，伺服器逐步推送）
- **是什麼**：一種「後端講一句、前端就收一句」的連線方式。不是等整段話講完才給你，而是邊講邊送。
- **作用**：生訊息草稿時，AI 的字會**一個一個浮現**（首字 < 0.5 秒就出現），不用盯著轉圈圈等 5 秒。
- **為什麼選它**：比一次回完整段更有「即時感」，使用者不會覺得卡；比另一種雙向連線（WebSocket）更簡單，我們只需要「後端→前端」單向推送就夠。

## 2. 後端 — 伺服器上的大腦

### FastAPI 0.115（後端框架）
- **是什麼**：後端的「總機 + 作業流程」。手機送來的每個請求（生草稿、記樣品、發問卷）由它接、分派給對應的處理邏輯、再把結果回給手機。
- **作用**：提供所有 `/api/v1/...` 端點、處理 SSE 串流、掛認證與合規。
- **為什麼選它**：用 Python 寫、速度快（async 非同步）、會**自動生成 API 文件**（前端工程師照著串就好）、型別檢查嚴謹，少出錯。

### Python 3.12（後端語言）
- **是什麼**：後端用的程式語言，可讀性高、AI/資料生態最強。
- **作用**：寫所有後端邏輯；AI 相關套件（LangGraph、Anthropic SDK）原生支援 Python。
- **為什麼選它**：AI 工具鏈幾乎都是 Python 優先；3.12 在速度與型別上都有改進。

### uv（後端套件管理）
- **是什麼**：管理「這個專案用到哪些 Python 套件、版本各是多少」的工具，順便建立乾淨的隔離環境。
- **作用**：安裝套件、鎖定版本，確保每個人、每台機器跑起來都一模一樣。
- **為什麼選它**：比舊工具（pip/poetry）快很多；也是本專案規則強制使用的工具（團隊一致）。

### APScheduler（排程器）
- **是什麼**：後端內建的「鬧鐘」。可以預約「48 小時後做某件事」。
- **作用**：發樣品後自動排 48h / 72h / 7d 三個跟進提醒；每天清晨產生「今日 5 件事」。
- **為什麼選它**：直接跑在 FastAPI 裡，Pilot 階段（11 人、低流量）不需要額外架一套複雜的排程服務，簡單夠用。

### Alembic（資料庫遷移）
- **是什麼**：資料庫的「版本控管 + 改建工班」。當資料表結構要改（加欄位、改型別），它幫你安全地、有紀錄地升級，不會把舊資料弄壞。
- **作用**：管理 27 張表的結構演進，每次改動都有可回溯的腳本。
- **為什麼選它**：FastAPI + SQLAlchemy 生態的標準搭配，成熟可靠。

### Pydantic（資料驗證）
- **是什麼**：後端的「門口檢查哨」。規定每個請求/回應該長什麼樣（哪些欄位、什麼型別），不合格直接擋下。
- **作用**：驗證手機送來的資料、定義 AI 服務的輸入輸出契約。
- **為什麼選它**：和 FastAPI 深度整合，「在系統邊界驗證輸入」是我們的硬規則，它做得最好。

## 3. LINE 整合 — 客戶從 LINE 傳訊進來、教練在 App 審核後回覆

> 這個模組是「客戶的入口」。客戶不需要裝任何 App，直接在 LINE 跟官方帳號說話就好；教練則在我們的 Care Copilot App 裡看訊息、看 AI 草稿、改完再按發送。

### line-bot-sdk + LINE Messaging API（LINE 官方 SDK）

- **是什麼**：LINE 官方提供的程式工具，讓我們的後端能「聽到」客戶傳來的訊息、也能把訊息「送回給」客戶。就像一條電話線，一頭接客戶的 LINE，另一頭接我們的系統。
- **在我們系統裡做什麼**：
  1. **收訊息**：客戶傳訊息給 LINE 官方帳號（OA）→ LINE 透過 webhook（見名詞速查）把訊息推給我們後端 → 後端解析、存入資料庫。
  2. **輪派教練**：新客戶第一次來，系統按照 A→B→C→A 的順序，自動把這位客戶分派給下一位教練，之後這個客戶就固定跟這位教練。Leader 可以看全部對話，也能手動改派。
  3. **生 AI 草稿 + 合規審查**：後端把訊息內容送給 AI 生一則「建議怎麼回」的草稿，草稿先過合規 sidecar 把關，沒問題才推給教練。
  4. **教練審核後發送**：教練在 App 看到草稿，可以直接採用或修改，按「發送」後，後端呼叫 LINE Messaging API 把訊息用 **push 方式**送到客戶的 LINE。
- **為什麼這樣設計（三個關鍵決策）**：
  - **AI 不自動回，教練按發送才算數**：直銷健康產品的對話很敏感，AI 草稿只是輔助，最終責任在人。自動回覆一旦說錯話、踩合規紅線，後果嚴重，所以我們硬性規定「人工確認才送出」。
  - **為什麼用 push 訊息，不用 LINE 即時回覆票（reply token）**：LINE 的即時回覆票大概 1 分鐘就失效，超過就沒辦法用。教練要看草稿、改草稿，幾乎一定超過 1 分鐘，所以改用 push 訊息（只要知道對方 LINE ID 就可以主動送，沒有時效限制）。push 訊息有每月免費額度，超量才另計費。
  - **為什麼要輪派（round-robin）**：一個 LINE 官方帳號多位教練共管，客戶進來不知道要找誰，輪派讓每位教練機會平均、新手不被跳過，Leader 也可以隨時調整。

### 訊息驗證（簽章驗證）

- **是什麼**：每則 LINE 傳進來的 webhook，我們後端會驗一個「防偽簽章」，確認這則訊息真的是 LINE 發的，不是壞人假造的請求。
- **為什麼要做**：外部 webhook 不驗就接，等於大門開著——任何人都能偽造訊息，讓系統亂發訊息或產生假資料。這是安全底線。

## 4. 資料庫與帳號 — 東西放哪、誰能看

### PostgreSQL 16 + pgvector（資料庫，自己跑的 container）
- **是什麼**：存放所有資料的「總倉庫」——客戶、互動、樣品、草稿、問卷……都放這。可以想成我們自己租了一間有保全的大倉庫，鑰匙在自己手上。開發時在本機起一個 container（像是桌上的小倉庫）；上線後放到 GCP 的雲端伺服器（同樣是 PostgreSQL，dev ↔ prod 切換順暢不費力）。
- **作用**：27 張資料表全部存這裡，保證資料正確、不遺失、查得快。
- **為什麼選它**：穩定、免費開源、功能強（支援 JSON、支援向量外掛 pgvector）；用標準 PostgreSQL container，想在哪跑都行，不被特定雲端平台綁住。

### pgvector（向量搜尋外掛，裝在 PostgreSQL 裡）
- **是什麼**：讓資料庫「看得懂語意」的外掛。它把文字變成一串數字（向量），就能找出「意思相近」的東西，而不只是「字面一樣」。
- **作用**：①活檔案語意搜尋（找「跟睡眠有關」的客戶，即使沒寫到「睡眠」兩字）；②合規詞庫變體攔截（「治療」改寫成「根治」也抓得到）。
- **為什麼選它**：直接長在 PostgreSQL 裡，不用再外接一套向量資料庫，架構更簡單、成本更低。

### RLS（Row Level Security，Postgres 內建的列級資料隔離）
- **是什麼**：資料庫層的「保全鐵閘」。規定「每一筆資料只有它的主人看得到」，由資料庫**強制**執行，不靠程式自律。這是 PostgreSQL 自帶的原生功能，不需要依賴任何外部服務。後端發 SQL 前，先帶上租戶身分（`SET LOCAL`），資料庫這層再強制驗一次——雙保險。
- **作用**：直銷商 A 永遠看不到直銷商 B 的客戶——這是我們的底線（跨租戶存取一律回「查無此資料 404」）。
- **為什麼選它**：把「資料隔離」這條最重要的安全規則下沉到資料庫，就算後端程式寫錯，資料庫這層也會再擋一次，雙保險。

### 自建 JWT 登入（後端自己發識別證）
- **是什麼**：「發識別證」的系統，由我們的 FastAPI 後端自己來做。使用者用 email + 密碼登入（密碼用 bcrypt 加密保存），後端發一張電子識別證（JWT）給你；之後每次請求都出示這張證，後端就知道你是誰、是哪個租戶、什麼角色。
- **作用**：直銷商 / Leader 兩種角色登入；識別證裡帶 `tenant_id`，讓 RLS 知道該放行哪些資料。
- **為什麼選它**：自己掌控帳號邏輯，不依賴外部服務；JWT 是業界通用標準，搭配 RLS 一條龍，帳號與資料隔離無縫接軌。

### Google Cloud Storage（GCS，語音檔暫存）
- **是什麼**：Google 的雲端檔案倉庫，專門放「不適合塞進資料庫」的大型檔案，例如語音。可以想成 Google 幫你租的一間有自動清潔服務的儲物間——語音檔放進去，7 天後自動消失，不用另外寫清除程式。
- **作用**：把訊息草稿轉成的語音檔（≤ 60 秒）暫存在 GCS；7 天自動刪除用 GCS 的生命週期規則設定，省去手寫排程。
- **為什麼選它**：整套系統已在 GCP 上，GCS 原生整合；生命週期規則直接在 GCS 設，幾個點選就搞定自動清除，乾淨省心。

### embedding（text-embedding-3-small，文字轉向量模型）
- **是什麼**：把一段文字「翻譯成一串數字」的小型 AI 模型，是 pgvector 能做語意搜尋的前置步驟。
- **作用**：把活檔案內容、合規詞庫轉成向量存進 pgvector。
- **為什麼選它**：體積小、便宜、夠用；搭配 pgvector 同庫儲存，架構不複雜。

## 5. AI — 系統的智慧核心（沿用既有框架）

> 這一層是「沿用既有框架（PRD 第 5 章），本次不重新設計」。以下說明它們各自是什麼、為什麼這樣分工。

### LangGraph 0.2（AI 編排）
- **是什麼**：AI 的「指揮家 / 流程圖」。一個任務常常要好幾個 AI 步驟（先判情緒、再生首字、再寫全文），LangGraph 負責安排這些步驟的先後與銜接。
- **作用**：指揮草稿生成流程——什麼時候叫 Haiku、什麼時候交給 Sonnet、中間怎麼串流出去。
- **為什麼選它**：把多步驟 AI 流程管理得有條理，且**包一層內部抽象**，避免被特定 AI 廠商綁死（將來要換模型比較容易）。

### Claude Haiku 4.5（速度型 AI 模型）
- **是什麼**：「反應快的 AI」。回答又快又便宜，適合不需要太深思考、但要馬上回的事。
- **作用**：情緒三檔判斷、異議接話、草稿「首字」、樣品跟進草稿、生活事件抽取——這些都求快。
- **為什麼選它**：延遲低、成本低；使用者最在意「會不會等太久」，快模型負責撐住即時體驗。

### Claude Sonnet 4.6（品質型 AI 模型）
- **是什麼**：「想得深的 AI」。回答更細緻、更像真人，適合需要品質的內容。
- **作用**：活檔案 insight、訊息草稿「全文」、問卷摘要、招募話術——這些求品質。
- **為什麼選它**：草稿「讀起來像朋友寫的」是整個產品最關鍵的一條（PMF 命脈），這種需要溫度與分寸的內容交給品質型模型。
- **快慢搭配的巧思**：訊息草稿先用 **Haiku 生首字（< 0.5 秒讓你看到開頭）**，同時 **Sonnet 在背後寫完整段**——又快又好，兼顧即時感與品質。

### 合規 sidecar（regex 規則引擎，非 AI）
- **是什麼**：貼在主流程旁邊跑的「自動校稿員」。用一份高風險詞庫（約 50 詞）逐字比對，毫秒級判斷紅/黃/綠燈。
- **作用**：**所有**要送給客戶的草稿（訊息/樣品跟進/語音/異議/招募）送出前都先過它；紅燈強制擋下並要求改寫，避免踩 FTC（收入保證）/FDA（療效宣稱）紅線。
- **為什麼選它（v1 先用規則、不用 AI）**：規則引擎**極快（< 50ms）、可預測、可稽核、零 AI 成本**。Pilot 先觀察誤判率，未來再加 AI 二審。合規是底線，寧可先用簡單可靠的方式守住。

### 太業務員警報（純規則引擎，非 AI）
- **是什麼**：另一個「貼邊小檢查」。盯著你對同一位客戶最近幾則訊息，連續 3 則都在賣東西就跳警告。
- **作用**：用產品關鍵字 + 連結樣式判斷「這是不是推銷訊息」，連 3 則就提醒「先寄一張純關懷再賣」。
- **為什麼選它（用規則不用 AI）**：判斷邏輯單純、要快、要省；Pilot 先觀察「會不會太常誤觸惹人煩」，再決定要不要做厚。

### TTS（OpenAI TTS 或 ElevenLabs，文字轉語音）
- **是什麼**：把文字「念出來」變成語音檔的 AI 服務。
- **作用**：把訊息草稿一鍵轉成語音（≤ 60 秒），直銷商下載後傳給客戶——亞洲市場很多人習慣聽語音。
- **為什麼選它 + 為什麼還沒定**：兩家各有優勢（成本 vs 音色），所以我們先用一層**抽象介面（VoiceProvider）**把它包起來，W4 前再決定用哪家——將來換供應商只改一個地方，不動其他程式。

## 6. 觀測與維運 — 看系統健不健康

### OpenTelemetry 1.x（追蹤標準）
- **是什麼**：系統的「行車記錄器」標準格式。記錄每個請求走過哪些步驟、各花多少時間。
- **作用**：追蹤 API 延遲、錯誤率、服務之間的呼叫關係。
- **為什麼選它**：業界中立標準，不綁特定廠商；之後要換監控後端也不用改程式。

### Langfuse 3.x（AI 專用觀測 + 成本計量）
- **是什麼**：專門盯「AI 用得如何」的儀表板。每次叫 AI 用了哪個模型、吃多少 token、花多少錢、草稿有沒有被採用，全記下來。
- **作用**：監看每日 AI 成本（我們有「每位直銷商每日 ≤ USD 0.30」的硬上限）、追蹤草稿採用率。
- **為什麼選它**：一般監控工具看不懂 AI；Langfuse 專為 LLM 設計，成本控管與效果追蹤都靠它。

### Sentry（錯誤監控）
- **是什麼**：系統的「報修專線」。程式一出錯，立刻把現場（哪一行、什麼情況）回報給工程團隊。
- **作用**：前端 + 後端的錯誤集中告警；連「學習紀錄寫入失敗」這種背景小事也會通知。
- **為什麼選它**：前後端都能用同一套；出問題第一時間知道，不用等使用者抱怨。

## 7. 為什麼整體這樣搭？（設計取捨小結）

| 取捨 | 我們的選擇 | 背後考量 |
| :--- | :--- | :--- |
| 要不要上架 App Store | 不上架，做 PWA | 一套網頁 iOS/Android 通用，4 週 Pilot 最省 |
| 資料庫自架 container vs 完全託管 | 自己跑 PostgreSQL container，部署在 GCP | 整套在 GCP 部署、用標準 Postgres 方便 dev↔prod 切換、未來成本彈性高；相較於全託管方案，掌控度更高也不被特定平台綁住 |
| 資料隔離靠程式 vs 靠資料庫 | 兩層都做（程式 + RLS）| 安全是底線，雙保險 |
| 帳號系統外包 vs 自建 | FastAPI 自建 JWT（email/密碼 + bcrypt）| 部署在 GCP、不需要額外引入外部帳號服務；自建讓帳號邏輯完全在掌控內 |
| 語音檔自管 vs 雲端儲存 | Google Cloud Storage（GCS）| 整套在 GCP，GCS 原生整合；生命週期規則一設，7 天自動清除，不用另寫排程 |
| 合規用 AI vs 用規則 | v1 先用規則（regex） | 快、可稽核、零成本、可預測 |
| AI 求快 vs 求好 | 兩個都要（Haiku + Sonnet 搭配）| 首字用快模型撐即時感，全文用好模型保品質 |
| 綁定某 AI 廠商 vs 保留彈性 | 包抽象層（LangGraph、VoiceProvider）| 將來換模型/換語音供應商只改一處 |
| LINE 回覆用即時回覆票 vs 用 push 訊息 | 用 push 訊息 | 即時回覆票約 1 分鐘就失效，教練審核草稿的時間幾乎一定超過；push 訊息沒有時效限制，教練改完隨時能發 |
| 新客戶誰來接：手動指派 vs 自動輪派 | 自動輪派（round-robin），Leader 可手動改 | 一個官方帳號共用多位教練，客戶進來時沒有「指定對象」；輪派確保機會平均、新手也有客戶，Leader 保留手動調整權 |

## 8. 名詞速查（最常被問的）

| 名詞 | 一句話 |
| :--- | :--- |
| **PWA** | 像 App 的網頁，可加到手機主畫面，免上架 |
| **SSE** | 後端逐字推、前端逐字收，讓草稿「浮現」 |
| **RLS** | PostgreSQL 內建的資料隔離，資料庫層強制「只有主人看得到」 |
| **JWT** | 登入後拿到的電子識別證，每次請求出示 |
| **pgvector** | 讓資料庫看得懂「語意相近」 |
| **embedding** | 把文字翻成一串數字，是語意搜尋的前置 |
| **sidecar** | 貼在主流程旁邊跑的小檢查器（如合規） |
| **token（AI）** | AI 計費與長度的單位，約等於「字的碎片」 |
| **LangGraph** | 安排多步驟 AI 流程的指揮家 |
| **租戶（tenant）** | 一個獨立的資料隔間（每位直銷商的資料互不相通） |
| **GCS** | Google Cloud Storage，Google 的雲端檔案倉庫，我們用來暫存語音檔 |
| **Cloud Run** | GCP 上跑後端 container 的服務，上傳程式碼後幫你自動啟動、擴容 |
| **LINE Messaging API** | LINE 官方提供的程式介面，讓外部系統能主動傳訊息給 LINE 用戶；我們用它把教練核准的草稿送到客戶的 LINE |
| **webhook（LINE webhook）** | LINE 的「自動轉發機制」：客戶一傳訊息，LINE 就立刻把這則訊息轉送到我們後端的指定網址，讓我們的程式能馬上收到並處理 |
| **push message（push 訊息）** | 一種不限時效的 LINE 訊息發送方式——只要知道對方的 LINE ID，任何時間都能主動送訊息。相對於有時效限制的「即時回覆票」，push 訊息讓教練有充裕時間審核草稿後再發 |
| **輪派（round-robin）** | 把新進客戶依順序分配給教練：第一位客戶給 A、第二位給 B、第三位給 C、第四位再給 A……確保每位教練機會均等，沒有人被跳過 |
