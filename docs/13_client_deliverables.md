## 1. CRM 狀態判斷邏輯

對應 [12_phase1_mvp.md M4](./12_phase1_mvp.md)。

**客戶不需要直接定義「10 種狀態名稱」**，而是給我們**狀態轉換的判斷邏輯**，狀態數量我們依此設計（可能是 6、8、或 10 種）。

### 需要客戶提供

- [ ] **客戶旅程的關鍵節點**：從第一次接觸到成交，會經過哪幾個業務階段？
- [ ] **每個階段的判定條件**：客戶做了什麼（或多久沒做什麼）算進入下一階段？
  - 例：「填完問卷」→「已填問卷」狀態
  - 例：「商談後 X 天未成交」→「需回訪」狀態
  - 例：「Y 天無互動」→「沉默」狀態
- [ ] **試用期定義**：是否有試用期？多久算「試用中」？
- [ ] **「成交」定義**：付款完成？開卡？第一次出貨？
- [ ] **「未成交」與「需回訪」差別**：客戶說 NO 跟暫時沒回應的處理有差嗎？
- [ ] **特殊狀態**：是否有「VIP 重點客」「需主管支援」等需特別處理的？

### 我方會做

- 依以上邏輯設計狀態機（合理範圍 6-10 種）
- 設計自動觸發規則（如：48h 未跟進 → 自動轉狀態）
- 設計手動可改 vs 系統自動的權限

### 範例：客戶可這樣交付

```
新名單 → 問卷送出後 → 已填問卷
已填問卷 → 教練約商談 → 已預約
已預約 → 商談完成 → 已商談
已商談 → 推薦商品 → 已推薦
已推薦 → 客戶試用 / 直接成交 / 拒絕
試用中 → 7 天回饋 → 試用回饋（好/壞）
成交 → 付款完成
未成交 → 客戶明確拒絕
需回訪 → 推薦後 7 天無回應、或客戶說「再考慮」
沉默 → 30 天無互動
```

格式不拘（流程圖、Excel、Word 都可），重點是**邏輯清楚**。

## 2. 問卷題目 + 計分規則 🔴

對應 [12_phase1_mvp.md M2](./12_phase1_mvp.md)。三階段問卷需完整版。

### Phase 1 — 快速分流（5 題）

- [ ] 5 題題目原文（中文）
- [ ] 答案選項（單選 / 多選 / 文字）
- [ ] **每題的健康關注標籤對應**（例：選 A → 標籤「壓力」）

### Phase 2 — 六大核心（12 題）

涵蓋：壓力、睡眠、消化、手腳冰冷、水腫、排便

- [ ] 12 題題目 + 答案選項
- [ ] **每題分值表**（0-3 分 per 題）
- [ ] **加總計分規則**（如：每類 2 題加總 → 該類分數）

### Phase 3 — 動態追問（3-8 題）

依 Phase 2 答案動態顯示：飲食、水分、活動、體態、女性週期、補充品/藥物

- [ ] 動態題庫（每類 2-5 題）
- [ ] **路由規則**：Phase 2 滿足什麼條件 → 顯示 Phase 3 哪幾題
  - 例：「壓力分數 ≥ 6」→ 顯示「壓力相關 5 題」

### 紅旗判定規則

- [ ] **觸發紅旗的答案組合**
  - 例：「胸痛 + 高血壓家族史」→ 紅旗
  - 例：「孕期 + 服用特定藥物」→ 紅旗
- [ ] **紅旗對教練的提示文字**（教練版摘要會顯示）

### 健康關注 Top 3 排序邏輯

- [ ] 怎麼從多個標籤中選出「最該關注的前 3 項」？
  - 例：「分數最高 + 紅旗加權」？
  - 例：「客戶自選 1 個 + 系統推 2 個」？

### 雙版摘要範本

對應 F2.5 / F2.6。

- [ ] **2-3 篇歷史摘要範例**（脫敏即可）
- [ ] **客戶版 vs 教練版** tone 差異標注（客戶版禁用診斷詞、教練版含紅旗 / SKU 推薦理由）

## 3. 商品庫（SKU）🔴

對應 [12_phase1_mvp.md M3 + Recommendation](./12_phase1_mvp.md)。

**Excel / Google Sheet 格式即可**，Pilot 期最少 20 個 SKU。

| 欄位 | 必填 | 範例 |
| :--- | :---: | :--- |
| `sku` | ✅ | `SYN-PROBIO-30` |
| `中文品名` | ✅ | 益生菌複方膠囊 |
| `主成分` | ✅ | 5 株益生菌、益生元 FOS |
| `針對問題標籤` | ✅ | 消化、便秘、免疫（對應 M2 健康標籤）|
| `建議用法` | ✅ | 每日 1 顆，餐後溫水服用 |
| `禁忌與注意事項` | ✅ | 免疫抑制患者請諮詢醫師 |
| `不建議組合` | ⚪ | 與抗生素間隔 2 小時 |
| `價格 NTD` | ✅ | 1,200 |
| `規格` | ✅ | 30 顆/瓶 |
| `產品文案 / 賣點` | ✅ | 50 字內賣點摘要 |
| `合規禁用詞` | ⚪ | 不得宣稱「治療便秘」 |

**重點**：「針對問題標籤」必須對應到 §2 問卷的健康標籤，AI 才能依問卷推 SKU。

## 4. 不合規詞彙（C1-C4）🔴

對應 [12_phase1_mvp.md 合規 AI](./12_phase1_mvp.md)、[03_adr.md ADR-017](./03_adr.md)。

由客戶法務 / 合規團隊提供 **CSV 檔**（不是 YAML），由我方匯入 admin 後台。

### 四類詞庫，每類 ≥ 40 條，總計 ≥ 200 條

| 類別 | 範例 | 量 |
| :--- | :--- | :---: |
| **C1 醫療宣稱** | 治療、治癒、預防 + 病名清單（糖尿病、高血壓、癌症...）| ≥ 60 |
| **C2 收入宣稱** | 月入 X 萬、保證收入、被動收入、財富自由... | ≥ 40 |
| **C3 誇大效果** | 100% 有效、立即見效、神奇... | ≥ 40 |
| **C4 金字塔風險語句** | 拉人頭、上線抽成、層級獎金... | ≥ 40 |

### CSV 格式

```csv
category,phrase,severity,suggested_rewrite
C1,治療糖尿病,high,支持血糖管理
C1,預防癌症,high,強化日常保健
C2,月入十萬,high,額外收入機會
C3,百分百有效,medium,有效成分
C4,拉人頭,high,推薦合作夥伴
```

- `severity`：`high` / `medium` / `low`
- `suggested_rewrite`：替代用語（系統自動改寫時用）

### 我方會做

- 匯入後系統自動計算 embedding（向量），用於攔截改寫變體（如「**治療**糖尿病」→「**根治**糖尿病」也會被抓到）
- 提供 admin UI 後續可線上補充 / 修改詞表（不需重新部署）

## 5. 話術庫

對應 [12_phase1_mvp.md M3](./12_phase1_mvp.md)。

| 類別 | 內容 | 量 |
| :--- | :--- | :---: |
| **開場話術** | 不同來源（LINE 邀約 / 活動 / 朋友介紹）的開場 | ≥ 5 種 |
| **產品銜接話術** | 從問卷痛點切到產品推薦的橋接句 | ≥ 10 條 |
| **異議處理庫** | 常見異議 + 標準回覆（「我再想想」「太貴了」「我老公不同意」等）| ≥ 15 對 |
| **柔性成交話術** | 不施壓的下單邀約 | ≥ 5 種 |
| **跟進話術範本** | 48h / 7d / 30d 三節點 | 每節點 ≥ 3 種 |

### 格式建議

每條話術標 3 個標籤便於 AI 檢索：

```
| 適用情境 | 對應 SKU | 關鍵字 |
| 客戶說「太貴」| 通用 | 價格、CP值 |
| 推薦益生菌時 | SYN-PROBIO-30 | 消化、便秘 |
```

格式不拘（Excel / Word / Google Doc），但**標籤要明確**。

## 7. 客戶**不需要**操心的事

以下技術項由我方規劃 / 採購 / 維運，**僅在最後階段需要客戶配合一次**：

| 項目 | 我方處理 | 客戶配合（如有）|
| :--- | :--- | :--- |
| GCP 帳號與部署 | ✅ 我方代開 + 維運 | 簽署月費對接（若客戶要自己 billing）|
| LINE OA 申請 | ✅ 我方協助申請流程 | 提供商家證明文件（W0 D1）+ 收驗證信 |
| WhatsApp Business 帳號 | ✅ 我方協助設定 | 提供商家證明 + 企業電話號碼 |
| Email 寄件網域 | ✅ 我方用 `synergy-ai.tw` 子網域 | 客戶若要用自家網域才需配合 DNS |
| Admin 帳號名單 | ✅ 我方建初始 admin | 確認 1 位客戶側窗口的 email |
| Pilot 教練人選 | — | ✅ 客戶指派 3-5 位（W0 D1）|
| Pilot 教練培訓 | ✅ 我方做 30 min onboarding | 客戶側 PM 出席 |
| 客戶資料儲存 | ✅ 我方 GCP Cloud SQL（加密）| 簽 DPA 資料處理協議 |

**Pilot 教練 3-5 位**仍需客戶指派（這個無法我方代決），其餘技術項客戶基本不用煩惱。
