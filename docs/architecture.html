0. 一眼看懂：三層鳥瞰

最簡單的理解方式：由上往下三層。使用者在前端操作，指令送到後端處理，資料與 AI 在底層支撐。每一句要送給客戶的話，都會先經過合規守門。

3. 資料流①　生訊息草稿（三道檢查 → 逐字串流）

這是最核心的流程，4 個工具接力：情緒感測太業務員合規草稿。重點：生成前先做三道檢查，AI 字逐個浮現（首字 < 0.5 秒），AI 絕不自動送出——由直銷商按「複製到 LINE」。

sequenceDiagram autonumber participant D as 直銷商 PWA participant API as 草稿串流端點 participant Em as 情緒感測 participant Sal as 太業務員檢查 participant Draft as 草稿服務 participant Comp as 合規 sidecar participant LG as LangGraph（沿用既有） participant AI as Claude Haiku→Sonnet participant DB as PostgreSQL D->>API: 要求生成草稿（contact、tone） API->>Em: 取最近情緒 Em-->>API: stressed（信心 0.87） Note over API: 壓力大 → 自動切「關懷」語氣 API->>Sal: 檢查連續推銷次數 Sal-->>API: streak = 2（未達 3，放行） API->>Comp: 前置輕量掃描 Comp-->>API: green（12ms） API->>Draft: 開始串流草稿 Draft->>LG: 呼叫草稿 agent LG->>AI: 生首字（Haiku ≤ 0.5s） AI-->>LG: 「Anna，」 LG-->>D: SSE：first_token（首字浮現） LG->>AI: 全文串流（Sonnet） AI-->>LG: token…token…token LG-->>D: SSE：token（逐字浮現） Draft->>Comp: 全文合規檢查 Comp->>DB: 寫入 compliance_checks Comp-->>Draft: green Draft->>DB: 寫入 message_drafts（green） Draft-->>D: SSE：compliance_result（綠燈） Draft-->>D: SSE：done（draft_id） Draft->>DB: 更新每日 AI 成本 Draft--)DB: 非同步寫 learning_logs Note over D: 直銷商按「複製到 LINE」（AI 不自動送） D->>API: 採用草稿（adopt）
紅燈異常路徑：若全文合規檢查回傳「紅燈」，後端推送 compliance_blocked，不寫入草稿、複製按鈕鎖住，直銷商必須改寫後重生——這是不可逾越的守門規則。

4. 資料流②　發樣品 → 自動排程 → 今日 5 件事跳卡

樣品轉換是精油品牌的命脈。直銷商只要按一下「我發了樣品」，系統自動排好 48h / 72h / 7d 三個跟進提醒，時間到就預生好草稿、推到今日 5 件事，不會錯過黃金跟進時機。

5. 資料流③　發問卷一次性連結 → 客戶免登入填 → 回填活檔案

客戶是唯一會接觸到的介面，也是唯一免登入的入口。連結 7 天有效、填一次就失效；填完 AI 生摘要（過合規），結果回填到客戶活檔案並推一張今日任務卡給直銷商。

sequenceDiagram autonumber participant D as 直銷商 PWA participant API as FastAPI 後端 participant Q as 問卷服務 participant Sonnet as Claude Sonnet participant Comp as 合規 sidecar participant C as 客戶（免登入） participant DB as PostgreSQL D->>API: 對客戶發問卷 API->>Q: 建一次性連結（7 天） Q->>DB: 寫 questionnaire_links（token） Q-->>D: 連結 URL D->>C: 透過 LINE / WhatsApp 傳連結 Note over C: 客戶點開（免註冊、免登入） C->>API: 開啟連結 API->>Q: 用 token 取題目 Q->>DB: 驗證未過期 / 未填過 Q-->>C: 12 題問卷 C->>API: 送出答案 Q->>DB: 標記連結已填、失效 Q->>Sonnet: 生成摘要 + 關懷切角 Sonnet-->>Q: 摘要 Q->>Comp: 摘要過合規 Comp-->>Q: green Q->>DB: 寫 questionnaire_responses Q->>DB: 回填 contacts（健康關注） Q->>DB: 新增 today_tasks 卡 API-->>C: 200「感謝填寫！」 Note over D: 直銷商收到通知 → 看摘要與切角

6. 資料流④　LINE 收訊 → 輪派 → 待回覆卡 → AI 草稿 → 教練審核 → push 回覆

每個租戶擁有一個LINE 官方帳號（OA）。客戶在 LINE 傳訊給 OA → LINE Platform 發 webhook 到後端 → 後端判斷新舊客戶（新客戶輪派給下一位 active 教練、既有客戶 sticky 維持原教練）→ 寫入 inbound_messages → 推送「待回覆」卡到今日 5 件事 → AI 自動生回覆草稿（過合規）→ 教練審核後按「發送」 → 後端呼叫 LINE Messaging API push 回覆給客戶。AI 不自動回覆，一切由教練把關。

寫 outbound_messages Note over Coach: AI 不自動回；共用 OA 用 push
新客戶輪派；既有 sticky
四大設計原則：① AI 不自動回覆——草稿必須教練審核後才能送出，合規紅燈時發送鍵鎖住。② 共用 OA——同租戶所有教練共用一個 LINE OA，以 push API 精準送給特定客戶。③ 新客戶輪派——依 official_accounts.assignment_cursor 取模輪派，確保工作量均衡；Leader 可在後台改派。④ 既有客戶 sticky——已有教練的客戶訊息直接路由回原教練，不重新輪派。非 OA 客戶（尚未加 LINE）的草稿維持「複製到 LINE」流程。

7. 多租戶隔離 — 為什麼 A 永遠看不到 B 的客戶

這是底線。隔離做兩層：後端中介層自動帶上 tenant_id，資料庫 RLS 再強制一次。就算後端程式寫錯，資料庫這層也會擋。跨租戶存取一律回 404（查無此資料），連「存在但無權」都不透露。

8. 登入流程 — 識別證怎麼運作

使用者登入後拿到一張電子識別證（JWT），裡面寫著「你是誰、哪個租戶、什麼角色」。之後每次請求都出示它，後端據此放行對應資料。認證由 FastAPI 後端自建，以 bcrypt 驗密碼、HS256 簽發 JWT，無須外部 Auth 服務。
