## 1. 合規紅線詞庫

### 背景說明

PRD 3.12 合規低語（精簡版）：v1 用 50 個高風險詞的規則庫（regex sidecar，< 50ms），所有外送草稿類型必須先通過此掃描。此詞庫是整個合規體系的唯一真實來源，優先於 AI 判斷。

### 阻塞分析

| 被阻塞模組/資料表/端點 | 阻塞原因 |
|---|---|
| `compliance_sidecar.py`（sidecars/） | 詞庫未定，sidecar 無法實作 regex pattern |
| `compliance_lexicon`（資料表） | `term`、`risk_level`、`suggestion`、`category`、`version` 欄位全部依賴客戶提供 |
| `compliance_checks`（資料表） | sidecar 未完成，觸發記錄無法寫入 |
| `POST /api/v1/compliance/check` | 端點實作依賴 sidecar |
| `message_draft_service.py`（T06） | 草稿生成流程三道前置檢查之一 |
| `voice_service.py`（T08） | 語音生成前文字必須先過合規 |
| `objection_service.py`（T09） | 異議回應通過合規 gate 後才放行 |
| `recruitment_service.py`（T11） | 招募話術走最嚴格合規 |
| `questionnaire_service.py`（T10） | 問卷摘要過合規 gate |

### 客戶需提供的具體資產

#### 1.4 需要客戶額外確認的事項

- [ ] Synergy 品牌方是否有自家比法規更嚴的禁用詞清單（超越 FTC/FDA 規定）
- [ ] 哪些詞彙在「品牌官方說法」下是合規的（正面清單，供 `brand_products.allowed_claims` 欄位使用）
- [ ] 法務 reviewer 姓名（填入 `compliance_lexicon.reviewed_by`）與 review 期限
- [ ] 詞庫版本控制策略（`compliance_lexicon.version` 欄位命名規則）

### 優先級與截止

| 項目 | 優先級 | 建議截止 | 驗收標準 |
|---|---|---|---|

| 法務 reviewer 簽署 | P0 | W1 結尾 | `compliance_lexicon.reviewed_at` 不為 null |
| 正面宣稱清單（各產品可說什麼） | P0 | W1 結尾 | 填入 `brand_products.allowed_claims` text[] |

### 遲交 Fallback

若 W1 結尾詞庫未到位：工程團隊使用 Synergy 官方訓練教材 + FTC/FDA 公開指南自行整理 20 個最明確紅線詞作為暫代版本，標記 `version = "v0.0-internal-draft"`，不對外展示；客戶正式詞庫到位後立刻替換並重新 scan 既有草稿記錄。此情形下 W1 Demo 只展示綠燈與黃燈，不展示紅燈攔截（因無真正法務背書）。

## 2. 常見異議庫

### 背景說明

PRD 3.9 快速異議處理器：10 種最常見異議，每種提供 3 種風格回應（共情型/提問型/邀請型）。這是 Amy 最依賴的「對話現場救援」工具，資料來自 Synergy 教練訪談。PRD 9 章標記狀態：🟢 訪談排程中。

### 阻塞分析

| 被阻塞模組/資料表/端點 | 阻塞原因 |
|---|---|
| `objection_service.py` | 無異議庫，AI prompt 缺少 few-shot 範例無法校準 |
| `objection_templates`（資料表） | `key`、`label_zh`、`description`、`sort_order` 欄位依賴客戶確認 |
| `objection_responses`（資料表） | 回應品質依賴 few-shot 範例 |
| `GET /api/v1/objection-templates` | 沒有 seed data 無法回傳 |
| `POST /api/v1/objection-responses` | AI 生成品質依賴範例 |

### 客戶需提供的具體資產

#### 2.1 10 種常見異議清單與真人回應範例

客戶需確認下列 10 種異議的「顯示名稱」、「情境描述」，並提供每種至少 1–3 句真人會說的接話範例（作為 AI few-shot 素材）。

| 異議 key（契約命名） | 建議繁中顯示名稱 | 客戶確認描述（情境） | 真人回應範例 A（共情方向） | 真人回應範例 B（提問方向） | 真人回應範例 C（邀請方向） |
|---|---|---|---|---|---|
| `spouse_oppose` | 先生/太太反對 | 另一半對 MLM / 直銷持懷疑或抵制態度 | 「我懂，我先生一開始也這樣，他主要是擔心……」 | 「請問他主要擔心的是時間還是收入部分？」 | 「要不要找一天我們三個一起喝個咖啡，讓他直接問？」 |
| `too_expensive` | 太貴了 | 對產品價格有疑慮，覺得性價比不夠 | 「我第一次看到價格也嚇到，後來算了一下每天其實⋯⋯」 | 「你覺得貴主要是跟哪個比較呢？是跟超市的還是之前用過的？」 | 「要不要先試試樣品，感受一下再說？」 |
| `try_first` | 我先試試看 | 有興趣但態度保留，不想承諾 | 「當然，先試試看是最直接的方式」 | 「你最想觀察的是哪個方面？我可以先提醒你留意」 | 「好，我先寄一份給你，14 天後你告訴我感覺怎樣」 |
| `family_not_support` | 家人不支持 | 除了另一半之外的家人（父母、兄弟姊妹）反對 | 「家人本來就會擔心，這是愛你的表現……」 | 「他們主要擔心的是什麼？時間、金錢還是對直銷的印象？」 | 「可以請他們一起來我們的健康分享聚會看看嗎？」 |
| `effectiveness_doubt` | 效果有存疑 | 不確定產品是否真的有效 | 「這個疑慮很合理，我當初也有，後來是……」 | 「你最想改善的是哪個狀況？我找找看有沒有類似的真實回饋」 | 「我可以介紹你認識一位用了三個月的朋友，直接問她？」 |
| `no_time` | 我沒時間 | 生活已經很忙，怕再多一件事 | 「我也是，所以我才用這個 App 幫我管，現在每天只要 10 分鐘……」 | 「你現在最忙的是哪一塊？主業還是家裡？」 | 「要不要先試一週看看，每天只做一件事就好？」 |
| `already_have_product` | 我已經在用別的牌 | 已有其他健康產品，覺得不需要換 | 「你現在用的是什麼？我也很好奇不同品牌的差異」 | 「你現在用了多久了？感覺哪個地方覺得還不夠？」 | 「要不要同時試一個月，自己對比感受一下？」 |
| `mlm_fear` | 對直銷 / MLM 有疑慮 | 擔心被說是直銷，或對 MLM 模式有負面印象 | 「我以前也這樣想，後來我發現我根本沒在招募，就只是自己用、分享……」 | 「你對直銷的印象是從哪裡來的？有沒有遇過讓你不舒服的經驗？」 | 「我帶你看看我每天怎麼用這個 App，你感受一下跟你想的一樣嗎？」 |
| `income_doubt` | 收入不確定 | 懷疑能不能真的賺到錢 | 「我一開始也沒想靠這個賺錢，就當省點購物費用……後來自然有人問我在用什麼」 | 「你對收入的期待是補貼日用品，還是想變成主要收入？」 | 「我可以讓你看看我真實的帳單，但我不保證你會一樣，因為每個人時間投入不同」 |
| `need_think` | 我再想想 | 模糊的推遲，沒有明確反對點 | 「當然，這是你的決定，沒有壓力」 | 「你主要還在考慮哪個部分？我幫你把問題釐清一下」 | 「我下週再和你說一聲，你到時候有任何問題都可以問我」 |

#### 2.2 需要客戶額外確認的事項

- [ ] 10 種異議的優先排序（哪 3 種最常遇到，放最前面）→ 填 `objection_templates.sort_order`
- [ ] 「家人不支持」和「先生/太太反對」是否合併或分開（台灣文化情境）
- [ ] 是否有 Synergy 品牌特有的異議（例如「Synergy 的報酬計畫複雜」）
- [ ] 「收入不確定」異議的回應是否需要特別嚴格走合規 gate（FTC 敏感）

### 優先級與截止

| 項目 | 優先級 | 建議截止 | 驗收標準 |
|---|---|---|---|
| 10 種異議清單確認（key + 顯示名稱） | P0 | W1 | 可 seed 到 `objection_templates` |
| 每種至少 1 組真人範例（3 風格） | P0 | W2 Day 1 | AI few-shot prompt 測試採用率 > 50% |
| 優先排序確認 | P1 | W2 | `sort_order` 填入 |

### 遲交 Fallback

若 W2 前真人範例未到位：工程團隊使用 PRD 情境 10 的範例（先生反對、共情/提問/邀請）作為暫代 3 組 few-shot，讓 AI 生成品質維持基本水準；客戶訪談結果到位後增補。

## 3. 健康問卷題目

### 背景說明

PRD 3.10 健康問卷：可選工具（非主流程），客戶主動表達「想知道適合什麼產品」時才使用。一次性連結 7 天有效，12 題左右，無需登入。PRD 9 章標記：❌ 待答（Op + 法務）。

### 阻塞分析

| 被阻塞模組/資料表/端點 | 阻塞原因 |
|---|---|
| `questionnaire_service.py` | 無題目無法實作 |
| `questionnaire_templates`（資料表） | `questions` jsonb 欄位依賴題目結構 |
| `GET /api/v1/questionnaire-templates` | 無 seed data |
| `GET /api/v1/questionnaire-links/{token}` | 問卷題目為空 |
| `POST /api/v1/questionnaire-links/{token}/submit` | 摘要 prompt 依賴題目維度 |

### 客戶需提供的具體資產

#### 3.1 題目清單（建議 12 題，不超過 15 題）

以下為建議架構，客戶需確認或修改題目、選項、關注點分流邏輯。

**題目格式規格**（對應 `questionnaire_templates.questions` jsonb 陣列）：

```json
{
```

## 4. 情緒判斷標註樣本

### 背景說明

PRD 3.3 語氣／情緒感測器（精簡版）：三檔分類（stressed 壓力大 / neutral 中性 / happy 開心）。PRD 9 章明確標記：❌ W2 前要齊（AI Architect 責任），但來源材料（去識別化訊息）需要客戶（Synergy 教練）提供。

### 阻塞分析

| 被阻塞模組/資料表/端點 | 阻塞原因 |
|---|---|
| `emotion_service.py` | Haiku 4.5 few-shot prompt 缺乏標註樣本，分類精度無法校準 |
| `emotion_readings`（資料表） | `emotion`、`confidence` 欄位需校準後才能評估準確率 |
| `POST /api/v1/contacts/{contact_id}/emotion/detect` | 模型 prompt 未完成 |
| `voice_service.py` | 語音生成前情緒 gate 依賴 emotion_readings |

### 客戶需提供的具體資產

#### 4.1 標註格式規格

客戶提供真實客戶訊息（去識別化），並由教練標上三檔情緒標籤。

**去識別化要求**：
- 所有人名改為 A/B/C
- 所有地址/電話/LINE ID 移除
- 保留語氣和措辭（這是核心資料，不能過度清洗）

**標註格式（CSV 或 Excel）**：

| sample_id | raw_text（去識別化） | label（stressed/neutral/happy） | annotator | note |
|---|---|---|---|---|
| e001 | 「最近工作好忙，覺得快爆炸了，妳說的東西我根本沒時間看」 | stressed | 教練 A | 明確表達壓力 |
| e002 | 「好啊，謝謝妳推薦，我來看看」 | neutral | 教練 A | 禮貌性回應，情緒中性 |
| e003 | 「哇！真的耶！好神奇！我先生也要試！」 | happy | 教練 B | 驚喜明顯 |
| ... | ... | ... | ... | ... |

**數量要求**：三個類別各約 65–70 則，共 200 則。建議分佈：
- stressed：70 則（真實比例可能較少，需刻意補充）
- neutral：80 則（最常見）
- happy：50 則

#### 4.2 採集建議

- 由 5 位教練各提供約 40 則（含多種來源：LINE 訊息貼文/FB 留言/面對面對話紀錄）
- 每則訊息長度：2–100 字為佳；太短（如「好」「嗯」）或太長（超過 200 字）標記為 edge_case，不計入主要訓練集
- 多位教練標同一則訊息時，以多數決為最終標籤；若有爭議，標記 `ambiguous=true`
- 台灣繁中為主，部分客家語/台語口語亦可納入（提升多元語境）

#### 4.3 同意與隱私

- 客戶需確認：這些去識別化訊息僅用於模型校準，不會被 Anthropic 用於訓練（Anthropic API 條款不用訓練）
- 原始訊息截圖不用提交，只需文字轉打
- 建議教練在提供前告知對方「會用匿名方式改善 AI 判斷準確度」（口頭同意即可，書面同意 optional）

| 項目 | 優先級 | 建議截止 | 驗收標準 |
|---|---|---|---|
| 200 則去識別化標註樣本 | P0 | W2 Day 1 | 格式正確可匯入；三類各 >= 50 則 |
| AI Architect 完成 few-shot prompt 調優 | P0 | W2 Day 3 | 內部測試集準確率 >= 75%（三分類，主觀指標） |
| 直銷商 Demo 前情緒感測器可用 | P0 | W2 結尾 | PRD 7.1 驗證項目：AI 判斷方向跟真人感覺一致 |

## 10. 招募漏斗階段定義

### 背景說明

PRD 3.11 招募漏斗（精簡版）：四階段（warm_list → exposure → invitation → signed）。PRD 強調這是「證明這是直銷工具而非普通 CRM」的核心差異，招募話術走最嚴格合規（FTC 收入宣稱）。

### 阻塞分析

| 被阻塞模組/資料表/端點 | 阻塞原因 |
|---|---|
| `recruitment_service.py` | 四階段業務定義不清，無法寫正確的階段轉換邏輯 |
| `recruitment_stages`（資料表） | `stage` enum 的業務說明需要客戶確認（何時算進入下個階段）|
| `PUT /api/v1/contacts/{contact_id}/recruitment-stage` | 前端選項文字與觸發合規的條件依賴業務定義 |
| `GET /api/v1/recruitment/funnel` | 漏斗統計意義需要客戶確認 |
| Sonnet 4.6 招募話術 prompt | 需要知道每個階段的「對話目標」才能生成合適話術 |

### 客戶需提供的具體資產

#### 10.1 四個招募階段的業務定義

對應 `recruitment_stages.stage` enum（契約命名不可改），請客戶確認或補充業務說明：

| stage（契約命名） | 建議繁中說明 | 「進入此階段」的判定條件 | 「推進到下一階段」的典型 行動 | 合規注意事項 |
|---|---|---|---|---|
| `warm_list` | 暖名單：有潛力但尚未接觸產品/事業機會的人 | 教練認識此人、有一定關係基礎，但對方對 Synergy 一無所知 | 在自然對話中分享個人健康體驗（非推銷） | 不要說「我要介紹你一個賺錢機會」，先建立關係 |
| `exposure` | 曝光：已接觸或試用過產品，對 Synergy 有初步了解 | 對方已試用樣品，或參加過健康分享聚會，或看過品牌介紹 | 分享更多真實體驗；邀請參加聚會或 Zoom 說明 | 分享「我的體驗」，不做收入宣稱 |
| `invitation` | 邀約：已被正式邀請考慮加入事業 | 教練已直接詢問對方是否有興趣了解加入事業的可能性，對方表示願意了解更多 | 安排一對一說明、或 Upline 三方對話；處理常見異議（先生反對、太貴等）| **FTC 紅線**：不可說「保證月入 X 萬」；說「有人做得不錯，每個人成果不同」 |
| `signed` | 簽約：已正式成為 Synergy 直銷商 | 對方已完成入會申請並繳清入會費 | 新人訓練、歡迎上線、開始使用 Care Copilot | 合規已過，正式夥伴關係 |

#### 10.2 需要客戶額外確認的事項

- [ ] 「暖名單」人數有沒有上限建議（Pilot 期每位教練大概帶幾位在招募中）
- [ ] 從「曝光」到「邀約」是否應有中間步驟（例：參加 Zoom 說明會是否算獨立步驟）
- [ ] Leader Linda 看到的「團隊漏斗統計」應包含哪些維度（各階段人數 + 本週新進人數 + 轉換率）
- [ ] 招募話術模板：教練是否有現成的「邀約話術」範例可以提供（供 Sonnet 4.6 few-shot 使用，同時過合規）

### 優先級與截止

| 項目 | 優先級 | 建議截止 | 驗收標準 |
|---|---|---|---|
| 四階段業務定義確認 | P1 | W3 Day 1 | `recruitment_service.py` 階段轉換邏輯完成 |

|---|---|---|

| **FDA DSHEA / 衛福部健食法**：療效宣稱 | 保健食品只能宣稱 structure/function，不可宣稱治療疾病 | `compliance_lexicon` health_claim、efficacy_claim 類別 |

## 14. LINE 官方帳號設定 + 共管教練名單

### 背景說明

CANON v2 新增 LINE 渠道整合：每個租戶擁有一個 LINE Official Account（OA）。客戶傳訊給 OA 後，後端 Webhook（`POST /api/v1/webhooks/line`）以 X-Line-Signature HMAC 驗證收訊，再依 `line_user_id` 找到或建立 contact，新客戶以 round-robin 輪派教練（`official_accounts.assignment_cursor`），既有 contact 維持 sticky 指派；訊息寫入 `inbound_messages`，同時在今日 5 件事產生「待回覆」卡（`today_tasks.source_type = inbound_reply`）。系統自動生成回覆草稿並通過合規 sidecar，教練審核後 `POST /message-drafts/{id}/send` 呼叫 LINE Messaging API push 訊息。

**資料表異動摘要**

| 資料表 | 新增/變更 | 欄位說明 |
|---|---|---|
| `official_accounts`（新增） | 全新資料表 | 每租戶一筆；含 `channel_id`、`channel_secret`、`channel_access_token`、`assignment_cursor`（輪派游標） |
| `inbound_messages`（新增） | 全新資料表 | 紀錄每則客戶傳入訊息；含 `line_user_id`、`message_type`、`raw_payload`、`contact_id`（FK）、`assigned_distributor_id` |
| `contacts` | 欄位擴充 | 新增 `line_user_id`（unique per tenant）、`source`（`line` / `manual` / …）、`assigned_at` |
| `message_drafts` | 欄位擴充 | 新增 `sent_at`、`delivery_method`（`line_push` / `manual`）、`line_message_id`（push 回傳）、`in_reply_to_inbound_id`（FK → `inbound_messages`） |

**跨工具依賴**

- 待回覆卡（`today_tasks.source_type = inbound_reply`）依賴 **T05 今日 5 件事**的任務生成邏輯
- 自動回覆草稿生成與合規掃描依賴 **T06 訊息草稿引擎** + **SYS 合規 sidecar**（`compliance_sidecar.py`）
- Leader 改派教練：`PUT /contacts/{id}/assignment`（已有端點，需確認權限 guard 支援 `leader` role）

### 阻塞分析

| 被阻塞模組/資料表/端點 | 阻塞原因 |
|---|---|
| `webhooks.py` | 未取得 Channel Secret 無法驗證 X-Line-Signature；Channel Access Token 未到位無法 push 回覆 |
| `line_client.py` | Channel Access Token 必要，否則 Messaging API 呼叫全數 401 |
| `assignment_service.py` | 需要共管教練名單（哪些 distributor_id 屬於此 OA 的輪派池）才能寫入 `assignment_cursor` 初始值 |
| `official_accounts`（資料表） | `channel_id`、`channel_secret`、`channel_access_token` 三個欄位全部依賴客戶提供 |
| `inbound_messages`（資料表） | Webhook 未完成，無任何資料可寫入 |
| `POST /api/v1/webhooks/line` | 缺 Channel Secret 即無法過 HMAC 驗證，無法上線 |
| `POST /message-drafts/{id}/send` | 缺 Channel Access Token 即無法呼叫 LINE push |

### 客戶需提供的具體資產

#### 14.1 LINE OA 帳號資訊

客戶（Synergy Op）需在 LINE Developers Console 完成以下設定後提供給工方：

| 項目 | 說明 | 客戶提供（待填） |

| 項目 | 優先級 | 建議截止 | 驗收標準 |

### 遲交 Fallback

1. **Webhook 驗簽**：以本地測試環境的假 Channel Secret 進行 HMAC 驗證邏輯開發與單元測試，`webhooks.py` 業務邏輯（找/建 contact、輪派、寫 `inbound_messages`、產生今日 5 件事卡）全部在 mock 模式下完成。
2. **LINE push**：`line_client.py` 實作 adapter 介面，開發期間以 log-only mock 替代真實 API 呼叫；介面相同，憑證到位後只替換設定值，不需改業務邏輯。
3. **輪派邏輯**：不依賴真實 OA，以 seed 資料（`official_accounts` 假一筆）驗證 `assignment_cursor` round-robin 行為。
4. **拿到真實憑證後**：填入 `.env`，切換 `line_client.py` 為真實模式，執行整合測試，W3 結尾完成端對端驗收。
